## ✅ 做對的部分

1. **職務定義清楚** — `/hr/roles` 頁面提供模組 × 分頁細粒度權限設定（can_read / can_write）
2. **新員工分配職務強制** — EmployeeForm 新增時必須選擇職務
3. **職務權限綁定** — 員工登入時從 role_id 查職務、再查權限
4. **權限實時生效** — role_tab_permissions 登入時自動簽入 JWT
5. **覆寫機制** — employee_permission_overrides 表支持個案例外

## ❌ 業務流程偏離 + 風險

### 1. 員工能自行修改自己的職務（高風險）

**位置**：`src/features/hr/components/create-store.ts` 的 `update()` 函數

**問題**：
```typescript
// updateEmployee() 無權限檢查
const { error } = await supabase
  .from('employees')
  .update(data)  // data 包含 role_id
  .eq('id', employeeId)
```

**風險**：
- 任何員工都能直接呼叫 `updateEmployee()`
- 修改自己的 `role_id` → 繞過整個職務系統
- 例：業務員工給自己改成「管理員」職務

**應該做**：
- API 層檢查：修改人有沒有「編輯員工」權限
- 前端層檢查：不准員工編輯自己的 role_id（只能管理員改）

---

### 2. 直接給員工權限的路徑無檢查（中風險）

**位置**：`PUT /api/roles/{roleId}/tab-permissions` 和 employee_permission_overrides API

**問題**：
- permission-overrides API（個別員工例外權限）無身分檢查
- 理論上任何員工都能為自己新增額外權限

**應該做**：
- 只有「有權限管理權限的人」才能操作（例如管理員）

---

### 3. 職務權限修改無即時通知（中風險）

**現象**：
- 管理員在 `/hr/roles` 改某職務的權限
- 既有員工要等到**重新登入**才生效

**應該做**：
- 權限變更時推送更新、或每日重簽 JWT
- 或在 JWT 中加 version 控制、變更時 increment

---

### 4. 預設職務欠缺（低風險）

**現象**：
- 新租戶建立時會自動建「管理員」「業務」「會計」「助理」
- 但老租戶（例如 Corner）可能沒執行過這個邏輯

**應該做**：
- 檢查並補齊所有租戶的預設職務

---

### 5. 權限預覽不清楚（低風險）

**現象**：
- 新增員工時選完職務、UI 沒顯示「該職務有哪些權限」
- 員工對自己的權限邊界不透明

**應該做**：
- 選職務後、右側加「該職務的權限預覽面板」

---

### 6. 管理員職務被鎖定（極低）

**現象**：
- `/hr/roles` 頁面對 is_admin=true 的職務禁用 toggle
- 無法修改管理員權限（UI 層鎖死）

**評價**：
- 設計正確（管理員權限不應該被修改）
- 但無提示說明為什麼不能改

---

## 對照「職務是身份卡」原則

William 的業務脈絡：**新增員工 → 選職務（給名牌）→ 系統用名牌識別他在每個地方的權限**

**現在的實裝符合度**：
- ✅ 職務定義清楚（名牌內容明確）
- ✅ 新員工分配職務（給名牌）
- ❌ 但員工能改自己的名牌（應該禁止）
- ❌ 名牌改了沒立刻生效（應該實時推送）
- ❌ 員工看不到自己的名牌（應該透明）
