# Venturo ERP 基礎設施模組盤點 2026-04-21

## 本批模組整體概念

這批是系統的基礎結構層，包括使用者進出口（登入/登出）、個人設定、實用工具與文件編輯。**settings / tools / design 三者定位有疊層**：settings 走使用者個人偏好（改密碼、外觀、菜單），tools 是開發測試用途（重設快取、列印樣板），design 是古老的設計工具入口（已轉向 brochures）。Dashboard 和 /（主頁根路徑）目前指向同一客體。office 是新增的協作文件編輯空間，用 Univer 套件實現試算表、文件、簡報。整體而言，這些頁面都是**半成品或測試性質**，許多有待打磨。

## 頁面清單

### 1. 根路徑 / （主頁）

**Route**: /  
**標題**: (無 H1，轉向 Dashboard)  
**做什麼**: 重定向至 Dashboard，使用者進入系統看到首頁儀表板  
**主要資料對象**: 無（純轉向邏輯）  
**主要動作**: 自動轉向 /dashboard  
**紅旗**: 無明顯

---

### 2. Dashboard 儀表板

**Route**: /dashboard  
**標題**: (由 DashboardClient 元件決定，讀不到具體內容)  
**做什麼**: 顯示系統總覽、近期活動、關鍵數據  
**主要資料對象**: 看不到具體（DashboardClient 是獨立元件）  
**主要動作**: 依儀表板小工具內容而定  
**紅旗**: 轉向外部元件，無法靜態分析；可能有外部相依

---

### 3. 登入

**Route**: /login  
**標題**: Venturo （品牌名）  
**做什麼**: 使用者輸入公司代號、帳號、密碼登入系統  
**主要資料對象**: 無 DB 查詢（純客端表單 + auth store 呼叫）  
**主要動作**: 送出表單 → 驗證登入 → 轉向首頁或上一頁  
**紅旗**:

- localStorage 存公司代號和帳號（security 考量）
- 密碼切換顯示邏輯用原生 input
- 記住我 30 天機制未見實作細節

---

### 4. 無權限頁面

**Route**: /unauthorized  
**標題**: 存取被拒  
**做什麼**: 使用者無權限訪問某頁時的降落頁，顯示拒絕訊息和回首頁按鈕  
**主要資料對象**: 無  
**主要動作**: 返回上頁、返回首頁  
**紅旗**: 無明顯

---

### 5. 設定 - 主頁

**Route**: /settings  
**標題**: 設定  
**做什麼**: 修改個人資料（含頭像、名字、郵箱等）、修改密碼、管理登出  
**主要資料對象**: employees (自己的員工記錄)、可能涉及照片上傳到 storage  
**主要動作**:

- 編輯個人資料 + 儲存
- 修改密碼 Dialog
- 登出
  **紅旗**:
- EmployeeForm 嵌套在 settings 頁，是否全量載入？
- 改密碼用原生 input，無現代密碼欄位元件
- 修改密碼後直接 reload，無提示

---

### 6. 設定 - 外觀

**Route**: /settings/appearance  
**標題**: 設定（Tab 導航）  
**做什麼**: 切換亮色 / 暗色主題  
**主要資料對象**: 無（存在 localStorage）  
**主要動作**: 選擇主題 → 儲存到本機快取  
**紅旗**:

- 提示文字說「未來會提供旅客端自己切換主題」，表示這功能還在考量階段

---

### 7. 設定 - 公司資訊

**Route**: /settings/company  
**標題**: 設定（Tab 導航）  
**做什麼**: 管理公司基本資料（名字、統編、銀行帳號、公司印章上傳等）、部門管理（若有開啟 departments 功能）、團控功能（若開啟）、旅行屬性（若開啟）  
**主要資料對象**: workspaces, departments, workspace_modules  
**主要動作**:

- 編輯公司欄位並儲存
- 上傳公司 logo 和四種印章到 storage
- 新增/編輯/刪除部門（需 departments 功能開啟）
  **紅旗**:
- 大量重複的 ImageUploadField 邏輯，有抽象化空間
- 部門管理、團控、旅行屬性都是條件式渲染，混在同一頁，頁面職責過多
- 權限檢查只看 isAdmin，沒有更細化的功能級權限

---

### 8. 設定 - Workspaces （多分公司）

**Route**: /settings/workspaces  
**標題**: Workspace 管理  
**做什麼**: 擁有平台管理資格的人建立和管理多個 workspace（分公司、分支），每個可設類型、建立首位系統主管、上傳大小章  
**主要資料對象**: workspaces, employees, profiles, workspace_channels  
**主要動作**:

- 新增 Workspace（需驗證代號唯一性）
- 建立首位系統主管帳號
- 顯示登入憑證（複製到剪貼板）
- 上傳大小章、切換 workspace 狀態
  **紅旗**:
- 流程複雜：新增 → 建立系統主管 → 建立公告頻道 → 嘗試建立 bot，步驟多易出錯
- 權限隔離在前端（「使用前端過濾實現資料隔離」），不應只依賴前端
- 登入憑證顯示後無加密，若有截圖分享風險

---

### 9. 設定 - Menu （菜單可見性）

**Route**: /settings/menu  
**標題**: 設定（無 Tab，獨立頁）  
**做什麼**: 使用者自訂側邊欄菜單顯示/隱藏，按功能模組分類（業務、財務、HR、系統）  
**主要資料對象**: employees (hidden_menu_items 欄位)  
**主要動作**:

- Toggle 各菜單項顯示/隱藏
- 儲存設定
- 重設為預設值（全顯示）
  **紅旗**: 無明顯

### 10. 設定 - Modules （模組授權）

**Route**: /settings/modules  
**標題**: 設定（無 Tab，獨立頁）  
**做什麼**: 系統主管啟用 / 停用租戶級的高級模組（會計、庫存、BI），設定到期日  
**主要資料對象**: workspace_modules, workspaces  
**主要動作**:

- 啟用模組 + 設定到期日期
- 停用模組
- 顯示模組狀態和功能清單
  **紅旗**:
- 模組到期邏輯不清：到期還能用嗎？前端只檢查 is_enabled 和 expires_at，沒看到實際的功能閘門
- MODULE_INFO 硬編碼在頁面，新增模組需改程式碼

### 12. 設定 - Receipt Test （收據列印測試）

**Route**: /settings/receipt-test  
**標題**: 收據列印測試  
**做什麼**: 針式打印機收據定位校正工具，調整 mm 級別的偏移，左側輸入、右側預覽、支援列印  
**主要資料對象**: 無（純前端模擬）  
**主要動作**:

- 輸入基本資料（買受人、統編、日期、經手人）
- 輸入明細（固定 7 行：摘要、數量、單價、備註）
- 調整位置微調（左、上偏移）
- 列印預覽和實際列印
  **紅旗**:
- **完全測試工具，不該上線**
- 硬編碼的預設值（永弘旅行社、王小明）
- 旋轉 180 度、transform 等佈局技巧，可維護性低
- 沒有對應的正式收據模組（只有測試）

### 13. Tools - Flight Itinerary （機票行程單）

**Route**: /tools/flight-itinerary  
**標題**: (無，Dialog)  
**做什麼**: 將線上訂票平台（Trip.com）機票憑證轉換為旅行社品牌風格的行程單，支援列印  
**主要資料對象**: 無 DB（純演算法轉換 + 列印）  
**主要動作**:

- 選擇不同的預設樣本資料
- 切換中英文
- 預覽和列印
  **紅旗**:
- **硬編碼的樣本資料**（SAMPLE_DATA_1），沒有上傳或編輯功能
- 是設計驗證工具，不是生產工具
- 實際業務流程應該是上傳 PDF → 自動辨識 → 轉換，現在完全手動

### 15. Tools - Reset DB （重設快取）

**Route**: /tools/reset-db  
**標題**: 清空本機資料庫  
**做什麼**: 開發人員清空瀏覽器 IndexedDB 快取（VenturoOfflineDB 和舊 venturo-db），強制重新同步  
**主要資料對象**: 無（客端快取操作）  
**主要動作**:

- 點擊按鈕確認刪除
- 進度訊息顯示（步驟 1-4）
- 成功後重新整理頁面
  **紅旗**:
- **危險操作，需要系統主管權限守護** ✓（已檢查 isAdmin）
- 成功後直接 reload，無法無縫繼續工作
- 如果在多 tab 狀態執行，可能導致資料不一致
- **未見完整的防誤觸機制**（沒有二次確認碼或進階驗證）

### 18. Calendar （行事曆）

**Route**: /calendar  
**標題**: (由 CALENDAR_LABELS 決定，推測「行事曆」或類似)  
**做什麼**: 企業行事曆，支援月/週/日視圖，員工可新增/編輯/刪除事件，擁有平台管理資格的人可跨 workspace 檢視  
**主要資料對象**:

- calendar_events
- workspaces （擁有平台管理資格的人過濾）
- employees （頻繁 lookup 名字）
  **主要動作**:
- 月 / 週 / 日 視圖切換
- 前月 / 今日 / 次月 導航
- 新增事件（Dialog）
- 編輯 / 刪除事件
- 拖曳事件改日期
- 生日名單 Dialog
- Workspace 篩選（擁有平台管理資格的人）
  **紅旗**:
- 大量 Hook 邏輯分散（useCalendarEvents, useCalendarNavigation, useEventOperations, useMoreEventsDialog），本地化程度高
- 依賴外部 calendar library（FullCalendar），SSR: false 會有 hydration issue
- 「生日名單」功能看不到實作，可能是空殼

### 19. Todos （待辦事項看板）

**Route**: /todos  
**標題**: (由 LABELS 決定，推測「待辦」或「工作看板」)  
**做什麼**: 看板式待辦管理（Kanban），卡片可拖曳跨欄，支援優先級（星級 1-5）、期限、指派、子任務、筆記、相關項目、搜尋、展開檢視  
**主要資料對象**:

- todos
- todo_columns （使用者自訂欄位，支援 system column）
- employees （指派人員查詢）
  **主要動作**:
- 新增/編輯/刪除卡片
- 拖曳卡片跨欄（自動更新 status）
- 拖曳欄位重新排序
- 新增/重命名/刪除自訂欄位
- 快速新增（欄位內快速輸入）
- 優先級調整（星級互動）
- 完成狀態切換
- 展開檢視（包括子任務、筆記、相關項目）
  **紅旗**:
- **code 複雜度很高**（近 1100 行），難以維護
- memo 化元件（TodoCardMemo）用於最佳化，但邏輯耦合
- 待辦和 column 的 status 對應有潛在混亂（mapped_status vs todo.status）
- 默認欄位檢索邏輯 `columns.find(c => c.mapped_status === 'pending') || columns[0]` 若沒符合的欄位會爆炸
- 未見權限細化（只有 visibility 欄位做簡單篩選）

### 21. Office - 編輯器

**Route**: /office/editor  
**標題**: (由文件名決定)  
**做什麼**: 使用 Univer 套件編輯試算表 / 文件 / 簡報，支援自動儲存、另存新檔、匯出 Excel  
**主要資料對象**: office_documents, tours （另存時可選擇儲存到旅遊團）  
**主要動作**:

- 編輯內容（Univer 引擎）
- 自動儲存（延遲 2s）
- 另存新檔（可選私人 or 旅遊團）
- 匯出 Excel
- 管理頁簽（mobile 側邊欄）
  **紅旗**:
- **Univer 是第三方大型套件，客製化空間有限**
- 動態載入三個巨大編輯器（SSR: false），初載會很慢
- 另存時若選「旅遊團」會直接導回列表，UX 奇怪
- 旅遊團關聯邏輯還在實驗階段

## 小結：紅旗分佈

| 類別         | 紅旗頁面                                      | 說明                                     |
| ------------ | --------------------------------------------- | ---------------------------------------- |
| **半成品**   | receipt-test, flight-itinerary, hotel-voucher | 測試工具，樣本資料硬編碼，無真正上傳編輯 |
| **廢棄路由** | /design, /settings/bot-line                   | 應移除或整合                             |
| **危險操作** | reset-db                                      | 有守護（isAdmin），但無二次驗證          |
| **複雜度高** | todos                                         | ~1100 行，邏輯分散，memo 化不清          |
| **權限不細** | settings/company, modules                     | 只用 isAdmin，無功能級細化               |
| **外部相依** | dashboard, design/new, office/editor          | 動態載入或轉向，靜態分析困難             |
| **前端隔離** | settings/workspaces                           | 資料隔離靠前端過濾，不安全               |
