# 人資模組地圖

## 模組概念

人資模組分成「管理者視角」（一般前綴）和「員工自助視角」（my- 前綴）兩條線。管理者透過員工編號、職務、薪資扣款等基礎設定掌控公司人事，員工透過打卡、請假、薪資條自查管理自己的工作狀態。核心流程是：行政設好打卡時間、假別類型、薪資項目 → 員工日常打卡申請假單加班 → 系統主管核准異常申請 → 薪資系統自動彙算。

## 頁面清單

### /hr

- **標題**：員工管理
- **做什麼**：管理者檢視公司在職員工列表（過濾已離職與 Bot），可編輯員工資料或辦理離職。
- **主要資料**：employees 表（display_name, employee_number, personal_info, job_info, status, hire_date）、roles 表（職務清單）。
- **主要動作**：新增員工、編輯員工、終止僱傭（Terminate）。
- **紅旗**：
  - 第 150 行註解「2026-04-18 統一過渡期」表示 role_id 欄位有新舊版本並存、暗示遷移未完成。
  - 職務列表從 API `/api/permissions/roles` 拉取，未來可能改成直接存在 employees。
  - 無明顯。

### /hr/announcements

- **標題**：公司公告
- **做什麼**：系統主管發布公司內部公告（分類為一般、制度、活動、緊急），員工可以看到帶有「置頂」標籤的重要訊息。
- **主要資料**：announcements 表（title, content, category, is_pinned, created_by, published_at）。
- **主要動作**：新增 / 編輯 / 刪除公告、置頂（系統主管限定）。
- **紅旗**：無明顯。

### /hr/attendance

- **標題**：出勤管理（委託至 Feature 元件）
- **做什麼**：管理者檢視所有員工的出勤紀錄並可能進行異常調整。
- **主要資料**：attendance_records 表。
- **主要動作**：查看、調整出勤狀態（實作細節在 `@/features/hr` 的 AttendanceManagementPage）。
- **紅旗**：頁面是包裝元件、無直接業務邏輯、需看 features 資料夾。

### /hr/clock-in

- **標題**：上班打卡
- **做什麼**：員工點擊「上班」「下班」按鈕記錄打卡時間，系統自動取得 GPS 位置、顯示即時台北時間、計算工時。
- **主要資料**：attendance_records 表（clock_in, clock_out, work_hours, overtime_hours, status）。
- **主要動作**：上班 / 下班打卡（POST `/api/hr/clock-in`）。
- **紅旗**：
  - 第 44 行「logger.warn('GPS 錯誤')」表示位置取得失敗不會阻擋打卡。
  - 無硬編碼但 GPS 寬限度未明確定義（僅有 「無法取得位置」字樣）。

### /hr/deductions

- **標題**：薪資扣款與津貼設定
- **做什麼**：系統主管設定公司薪資扣款項目（勞保、健保、退休金等）和津貼類型（伙食、交通），支援固定金額、費率、級距三種計算方式。
- **主要資料**：payroll_deduction_types、payroll_allowance_types 表。
- **主要動作**：新增 / 編輯 / 刪除扣款項目、新增 / 編輯 / 刪除津貼項目、載入台灣預設設定。
- **紅旗**：無明顯。

### /hr/training

- **標題**：數位培訓
- **做什麼**：框架頁面，目前只顯示功能預覽與規劃清單，尚未上線完整的課程管理與進度追蹤。
- **主要資料**：無實際資料。
- **主要動作**：無。
- **紅旗**：
  - 第 9-11 行註解「第一版（靜態框架）」「未來會接入課程管理系統」表示功能未開發。
  - 整頁都是虛擬卡片示意。

### /hr/my-payslip

- **標題**：我的薪資條
- **做什麼**：員工查看歷月薪資明細，包括底薪、津貼、扣款、實發金額及出勤摘要。
- **主要資料**：payroll_records 表（base_salary, overtime_pay, bonus, meal_allowance, transportation_allowance, unpaid_leave_deduction, late_deduction, allowance_details, deduction_details, gross_salary, net_salary）、payroll_periods 表（year, month, status）。
- **主要動作**：按月份查看薪資（年份可變更）。
- **紅旗**：
  - 第 79-85 行有新舊欄位並存：allowance_details / deduction_details（新）VS meal_allowance / transportation_allowance（舊），code 中寫了 fallback 邏輯應對過渡。
  - 第 252 行 Row 元件無法處理負值顯示邏輯（負值在計算端已加負號、元件端須配合）。

### /hr/roles

- **標題**：職務管理
- **做什麼**：系統主管定義職務角色與細粒度權限（可讀 / 可寫），支援模組級與分頁級雙層設定。
- **主要資料**：roles 表、modules、tabs（權限定義） 。
- **主要動作**：新增角色、設定權限（模組級全開 / 全關 + 分頁級逐項、部分開啟用橘色圓點表示）、刪除角色（沒有系統主管職務的 角色）。
- **紅旗**：
  - 第 244 行 `logger.log('[ROLES] PUT payload ...)` 過度詳細的日誌（生產環境應移除或改 debug 級）。
  - 第 268 行 POST 後再 fetch 驗證欄位「確保 DB 狀態反映到 UI」表示對資料一致性有信心問題。

### /hr/my-attendance

- **標題**：我的出勤
- **做什麼**：員工檢視本月出勤紀錄，包括打卡時間、工時、加班時數、遲到狀態。
- **主要資料**：attendance_records 表（date, clock_in, clock_out, work_hours, overtime_hours, status）。
- **主要動作**：按年月篩選、查看日期清單。
- **紅旗**：無明顯。

### /hr/missed-clock

- **標題**：補打卡申請
- **做什麼**：員工申請補打卡（指定日期、類型為上班或下班、填寫實際時間與原因），系統主管核准或駁回；核准時系統會更新出勤紀錄。
- **主要資料**：missed_clock_requests 表（employee_id, date, clock_type, requested_time, reason, status）、employees 表（用於選項拉取）。
- **主要動作**：申請補打卡（員工）、核准 / 駁回（系統主管）。
- **紅旗**：
  - 第 100-130 行找系統主管流程冗長（透過 roles.is_admin、再撈員工、再篩選 job_info->role_id），表示「誰擁有管理員資格」的定義複雜（有 is_admin flag 又有 role_id）。
  - 第 114 行注釋「透過 roles.is_admin 判斷」表示 is_admin 欄位是單一真相來源但實作細節散落。

### /hr/payroll

- **標題**：薪資計算與發放（委託至 Feature 元件）
- **做什麼**：系統主管或人事人員管理薪資期間、計算薪資、生成薪資單並標記已確認 / 已支付。
- **主要資料**：payroll_periods、payroll_records 等（實作細節在 `@/features/hr` 的 PayrollManagementPage）。
- **主要動作**：期間設定、薪資計算、核准、發放（實作細節在 features）。
- **紅旗**：頁面是包裝元件、無直接邏輯。

### /hr/overtime

- **標題**：加班申請
- **做什麼**：員工申請加班（指定日期、時間範圍、自動計算小時數、可填寫原因），系統主管核准或駁回。
- **主要資料**：overtime_requests 表（employee_id, date, start_time, end_time, hours, reason, status）。
- **主要動作**：申請加班（員工）、核准 / 駁回（系統主管，同時可能更新薪資計算）。
- **紅旗**：
  - 第 96-98 行手工計算加班小時數（`endParts[0] - startParts[0] + ...`），如果跨午夜會算錯（但實務上加班通常不跨午夜）。
  - 第 112-126 行同樣冗長的找系統主管流程（與 missed-clock 重複）。

### /hr/my-leave

- **標題**：我的請假
- **做什麼**：員工檢視本年請假紀錄、餘額卡片（分假別展示已用 / 剩餘 / 額度），可按假別查看假額額度進度條。
- **主要資料**：leave_requests 表（employee_id, leave_type_id, start_date, end_date, days, reason, status）、leave_types 表（name, code）、leave_balances 表（entitled_days, used_days, remaining_days）。
- **主要動作**：查看紀錄與餘額（無新增、編輯、刪除權限在此頁）。
- **紅旗**：無明顯。

### /hr/leave

- **標題**：請假管理（委託至 Feature 元件）
- **做什麼**：系統主管或人事人員管理假別定義、期間設定、員工請假申請核准、餘額初始化與扣減。
- **主要資料**：leave_types、leave_requests、leave_balances（實作細節在 `@/features/hr` 的 LeaveManagementPage）。
- **主要動作**：設定假別、初始化餘額、核准 / 駁回申請（實作細節在 features）。
- **紅旗**：頁面是包裝元件。

### /hr/reports

- **標題**：出勤月報
- **做什麼**：人資或主管檢視全公司月度出勤彙總，包括員工總人數、總工時、總加班、總遲到，以及逐員工出勤統計表（出勤天數、正常 / 遲到 / 缺勤 / 請假天數、工時、加班）。
- **主要資料**：attendance_records 表（aggregated by employee_id）、employees 表。
- **主要動作**：按年月篩選、查看統計表。
- **紅旗**：無明顯。

### /hr/settings

- **標題**：人資設定
- **做什麼**：系統主管設定打卡工作時間、遲到寬限、GPS 定位、LINE 打卡機器人連線等。
- **主要資料**：workspace_attendance_settings 表（work_start_time, work_end_time, late_threshold_minutes, standard_work_hours, require_gps, gps_latitude, gps_longitude, gps_radius_meters, enable_line_clock, enable_web_clock）。
- **主要動作**：修改打卡設定、LINE 機器人連線設定（多步驟：建立 OA → 啟用 Messaging API → 複製 Token & Secret → 貼上並完成）。
- **紅旗**：
  - 第 112 行 API `/api/line/setup` 的 GET 與 POST 實作假設 API 會自動管理 line config 狀態，但沒有驗證邏輯。
  - 多個 fetch 呼叫無串連錯誤處理（第 177-186 行三個 POST 中任一失敗不會通知使用者）。
  - LINE 設定步驟複雜（4 步），且表單欄位無驗證（只檢查非空）。

## 共通紅旗與模式

1. **過渡期欄位並存**（payslip 的 allowance_details / meal_allowance、roles 的 role_id top-level vs nested）：表示 schema 改動未全面完成，code 中有 fallback 邏輯，應統一方案。

2. **找系統主管流程重複**（missed-clock 與 overtime）：透過 `roles.is_admin` + `employees job_info->role_id` 雙層查詢，應抽成共用 helper。

3. **Feature 元件包裝**（attendance、payroll、leave）：三頁只是路由層，業務邏輯在 features 資料夾，跨檔案審核成本高。

4. **API 呼叫無例外處理鏈**（settings 的 LINE 設定三個 POST）：若中間某一步失敗，UI 不會通知、用戶看不出發生什麼事。

5. **過度詳細日誌**（roles 的 logger.log 整個 payload）：生產環境可能洩露敏感資料，應改 debug 級。

## 員工 vs 管理者視角拆分

拆分方式很乾淨，約定俗成：

- **my-\*** 前綴（my-payslip、my-attendance、my-leave）= 員工自助、無核准權、只看自己的資料。
- **非 my- 前綴**（announcements、attendance、leave、payroll、reports、overtime、missed-clock、roles）= 管理介面。
  - 部分頁面（overtime、missed-clock）兼具員工申請 + 系統主管核准，透過 `isAdmin` flag 條件式渲染操作欄。
  - 公告只有 系統主管看得到新增按鈕，但所有人都看得到內容。

此拆分清晰、無交叉混淆，適合多角色權限管理。
