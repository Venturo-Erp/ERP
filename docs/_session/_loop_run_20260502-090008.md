### DB 抽查
新 partial index 存在：
```
attractions | idx_attractions_workspace_active | (workspace_id, is_active) WHERE (is_active = true)
customers   | idx_customers_workspace_active   | (workspace_id, is_active) WHERE (is_active = true)
```

### EXPLAIN — 誠實的觀察
跑了 attractions / customers 的典型列表 query（`WHERE workspace_id = X AND is_active = true LIMIT 50`），planner 仍選**舊**的 `idx_xxx_workspace_id`、不是新 partial index。

**這是 cost-based 的正常結果**：
- customers 385 row、attractions 2444 row、planner 估算 cost 接近、選誰都行
- 等資料量上來、planner 會優先選 partial（更小、命中率更高、寫入維護更便宜）
- partial index 是「未來投資」、不是當下效能優化

寫入端的好處立即生效：軟刪除（`is_active = false`）的 row 不會進這個 index、未來資料量大時 INSERT/UPDATE 開銷更低。
