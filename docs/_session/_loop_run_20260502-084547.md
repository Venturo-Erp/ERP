## 範圍決策（與 backlog 描述的差異）

backlog 列了 16 個函數（含 `update_updated_at_column` 等）。實際做下去發現：

1. **`update_updated_at_column` 不能 DROP** — 憲法 §16 凍結模組（`channels` / `messages` / `channel_groups`）的 trigger 仍綁這個 function、屬鐵律不能動。所以這個 function 跟它的所有 trigger 一律不動，等凍結期解除後另案清。
2. **3 個 backlog 列的 function 已不存在** — `update_payment_requests_updated_at` / `trigger_payment_requests_updated_at` / `suppliers_updated_at_trigger` 在 DB 裡不是 function 而是 trigger 名稱（backlog 寫法有混淆）。
3. **payment_requests / suppliers duplicate trigger 不在本次範圍** — 這是 B-002（前置 = B-001）的事，本次刻意只把 trigger function reference 改掉、不砍 duplicate。

最終本次只動 12 個確認 body 等價於 `set_updated_at()` 的變體 function，跟它們綁的 15 條 trigger。

**4. 等價性論證（替代 UPDATE row 抽查）**

避開 production data 改動、改用 function body 對比驗證：12 個變體 function 用 `pg_get_functiondef` 撈出來的 body 全部都是
```
BEGIN NEW.updated_at = now(); RETURN NEW; END;
```
（部分用 `NOW()` 大寫，PostgreSQL 大小寫不敏感、語意完全一致）。`set_updated_at()` body 也是相同。trigger metadata（timing=BEFORE、event=UPDATE、level=ROW）保持不變。所以 UPDATE 行為跟之前 100% 等價、不需實 UPDATE。
